# Один хід сліпого лупа

Ти працюєш у репозиторії **url-shortener**. Це **один хід** сліпого лупа: попередніх ітерацій
ти не пам'ятаєш і не пам'ятатимеш цієї. Увесь стан, який мусить пережити хід, лежить **на
диску** — git, `tasks/tracker.md`, `loop/JOURNAL.md`. Прочитай його і зроби **рівно один крок
уперед**.

Секцію «Ціль цього прогону» дописав ранер — вона внизу цього файла. Вона єдина міняється
між ходами.

## Читай першим, у цьому порядку

0. **Стан від попередньої ітерації.** Його вже влив тобі в контекст хук на початку сесії:
   спершу факти, які він щойно виміряв сам, далі журнал попередніх ходів. Немає нічого —
   отже ти перша ітерація.
1. [AGENTS.md](../AGENTS.md) — команди, золоті правила, чого не робити.
2. [docs/architecture-map.md](../docs/architecture-map.md) — конвенції, яких мусить триматись
   будь-яка зміна: домен проти HTTP, форма помилки, статус-коди, порядок роутів, міграції,
   заборона нових залежностей без ADR.
3. Ціль цього прогону (секція внизу) — і все, на що вона посилається.

## Протокол ходу

Візьми з трекера цілі **першу** задачу зі статусом `todo`, усі залежності якої вже `done`.
Жени її циклом `RED → GREEN → REFACTOR → GATE → REVIEW → COMMIT` за протоколом
[.claude/skills/implement/SKILL.md](../.claude/skills/implement/SKILL.md) (не-Claude агенти
читають ту саму копію в [.agents/skills/sdd-implement/SKILL.md](../.agents/skills/sdd-implement/SKILL.md)).

Цей prompt навмисно перевизначає стандартний single-agent режим skill. Ти — coordinator:
**не пиши тест або продукт-код сам**. У Claude Code послідовно викликай project agents за bare
`subagent_type`; не запускай їх паралельно й не створюй worktree. Не використовуй
general-purpose або single-agent fallback. Якщо іменованої ролі немає, запиши це в журнал і
заверши хід без коміта й без зміни tracker.

1. **RED → `test-author`.** Передай йому id задачі, task-файл, AC, DoD і `files_hint`. Він пише
   лише тест, запускає його й повертає команду, класифікацію GOOD/BAD RED та точний failing
   assertion. На BAD RED або false-pass повторно виклич `test-author`; не переходь до GREEN без
   GOOD RED.
2. **GREEN + REFACTOR + GATE → `implementer`.** Передай той самий task brief і RED-handoff.
   Він пише мінімальний продукт-код, не послаблює тест і запускає
   `npm run lint && npm run test:fast && npm run links:check`. E2E запускає лише там, де цього
   прямо вимагає DoD задачі.
3. **REVIEW → `reviewer`.** Після зеленого handoff передай йому task, AC і поточний diff.
   Reviewer працює read-only та повертає cited findings або `REVIEW_CLEAN`. Finding про тест чи
   непокритий AC поверни `test-author`, а потім `implementer`; finding про реалізацію поверни
   `implementer`. Після виправлення повтори gate й знову виклич `reviewer`.
4. **COMMIT → coordinator.** Лише після `REVIEW_CLEAN` онови `tracker.md`: задача → `done` і
   створи **один коміт на задачу** з трейлерами `SDD-Task: <id>` та `SDD-AC: <перелік>`.

## Заборонено

- Писати код до червоного тесту.
- Писати тест або продукт-код із coordinator-контексту замість dispatch іменованої ролі.
- **Послаблювати чужий тест, щоб він позеленів.** Не проходить, а ти певен, що код правильний, —
  ескалюй: критерій приймання може бути хибним. Не роби тест м'якшим.
- Вигадувати рішення, якого немає в специфікації. Не знаєш — не вгадуй (див. нижче).
- Комітити з червоним гейтом.
- `git push`. Стеля цього лупа — коміт у локальну гілку.
- `git checkout main`, `git switch main`, будь-яке перемикання гілки. Ти вже на потрібній.
- Редагувати `docs/roadmap.md`. Чергу веде людина.
- Додавати npm-залежність, якщо ADR цієї фічі прямо її не дозволив.

## Якщо ти заблокований

Задача впирається в питання, на яке ти не маєш права відповісти сам (наприклад `link-expiry`
T2 чекає на дефолтний TTL зі `spec.md` §8):

1. постав задачі в `tracker.md` статус `blocked`;
2. запиши там-таки **питання** — одним реченням, з посиланням на місце, де бракує відповіді;
3. заверши хід **без** `DONE`.

Не вгадуй. Наступна ітерація не зрушить метрику, і зупинка «немає прогресу» зупинить луп за
дві ітерації. Тоді відповість людина — а це швидше, ніж відкочувати вгадане.

## Останній крок ходу — запиши, що сталось

Наступна ітерація не пам'ятатиме цього ходу. Вона прочитає `loop/JOURNAL.md` — і більше нічого
від тебе не почує. **Допиши** в кінець файла (ніколи не переписуй його):

```markdown
### Ітерація <N> — <id задачі>
**Зробив:** одне речення про те, що тепер зелене.
**Спіткнувся:** що зламалось і чому. Порожньо — тільки якщо справді нічого.
**Наступному ходу:** попередження, яке ти хотів би прочитати сам.
```

Номер ітерації стоїть у секції «Ціль цього прогону» нижче. Пиши коротко: цей файл вливається
в контекст **кожної** наступної ітерації.

Пиши це і тоді, коли хід не дав прогресу. Особливо тоді: ітерація, яка мовчки нічого не
зробила, змушує наступну повторити ту саму помилку.

⚠ Журнал — твій **самозвіт**, і в цьому його слабкість. Поруч із ним хук показує факти з git і
трекера. Розходяться — вір фактам і напиши про розбіжність.

Файл у `.gitignore`, тож у коміт задачі він не потрапить. Ворота його не перевіряють — пиши
вільно.

## Коли створювати DONE

Тільки коли виконано **обидві** умови:

- усі задачі трекера цілі мають статус `done`;
- `npm run lint && npm run test:fast && npm run links:check` зелений.

Тоді створи **порожній файл `DONE`** у корені репо:

```bash
node -e "require('fs').writeFileSync('DONE','')"
```

⚠ Не `touch` — на Windows такої команди немає, і хід, який «завершився», насправді не
завершить луп.

Лишилась хоч одна `todo` — **не створюй `DONE`**. Просто заверши хід: наступна ітерація
підхопить наступну задачу.
