# Луп — ralph над однією фічею

Ранер, який жене SDD-пакет фічі без людини за кермом: бере ціль, віддає її агентові в
headless-режимі, після кожного ходу перевіряє детермінований гейт і зупиняється, щойно стає
марно.

**Гілку створюєш ти.** Ранер у git не пише взагалі — він відмовляється працювати на `main`
(`exit 2`) і на цьому його стосунки з git закінчуються.

Три файли, і всі три читаються за п'ять хвилин:

| Файл | Що це |
|---|---|
| `ralph.mjs` | рушій: гейт, три зупинки. Чистий Node, без npm-залежностей; спільні хелпери бере зі [`scripts/lib.mjs`](../scripts/lib.mjs) |
| `PROMPT.md` | статичний промпт одного ходу. Ранер дописує до нього секцію з ціллю й номером ітерації |
| `JOURNAL.md` | пам'ять між ітераціями. Пише агент, читає [хук](../.claude/hooks/loop-memory.mjs). У git не потрапляє |

```bash
git checkout -b feat/qr-codes                  # гілку створюєш ти, не скрипт
npm run ralph -- --dry-run                     # димова: що є в репо; 0 токенів
npm run ralph -- --feature qr-codes --dry-run  # ціль, метрика; 0 токенів
npm run ralph -- --feature qr-codes            # поїхали
```

Ціль задається **тільки** через `--feature <slug>`. Ранер не вгадує, що робити далі: черга —
рішення людини, і живе воно в [docs/roadmap.md](../docs/roadmap.md), який читають очима.

## Чому цикл сліпий

Щоходу агент отримує **один і той самий** промпт зі свіжим контекстом. Він не пам'ятає
попередню ітерацію — він її **читає**: `git log`, `tasks/tracker.md`, `JOURNAL.md`. Усе, що
має пережити хід, лежить на диску.

Це не обмеження, а конструкція. Довгий діалог гниє: модель тягне за собою власні хибні
висновки з дванадцятого повідомлення до сорокового. Сліпий хід не має чого тягнути. Ціною є
дисципліна: якщо стан не записаний на диск, його не існує.

## Пам'ять між ітераціями

Сліпота має вартість. Ітерація, яка витратила двадцять хвилин на те, щоб з'ясувати, що
`QRCode.toString` віддає Promise, забирає це знання з собою в могилу — і наступна витрачає ті
самі двадцять хвилин.

Тому пам'ять. Одне записує агент, друге рахує хук, і вони навмисно **не змішуються**.

**Агент пише журнал.** Останнім кроком ходу, за вимогою `PROMPT.md`, він дописує в `JOURNAL.md`:

```markdown
### Ітерація 1 — T1
**Зробив:** renderQrSvg + unit-тест.
**Спіткнувся:** toString віддає Promise, перший тест ловив [object Promise].
**Наступному ходу:** роут теж стане async.
```

**Хук читає і додає факти.** `.claude/hooks/loop-memory.mjs` спрацьовує на події `SessionStart`,
і все, що він друкує в stdout, Claude Code додає в контекст сесії. Хук нічого не кешує: гілку,
чистоту дерева, три останні коміти і статуси трекера він рахує **у момент читання**.

```
## Факти (хук щойно виміряв — їм вір)

гілка `feat/qr-codes` · робоче дерево чисте
трекер qr-codes: T1 done · T2 todo · T3 todo · T4 todo · T5 todo

## Журнал попередніх ходів (самозвіт агента — звіряй із фактами вище)
...
```

Різниця між двома блоками — це вся суть. Журнал каже «зробив T2», трекер каже `T2 todo`:
розбіжність видно з першого погляду, і промпт велить вірити фактам. Самозвіт допущено рівно
тому, що ніхто, крім моделі, не знає, на чому вона спіткнулась, — але його ніколи не лишають
без нагляду.

Три деталі, які легко пропустити:

- **Хук мовчить, коли журналу немає.** У чистому клоні звичайна інтерактивна сесія його навіть
  не помітить. Луп створює журнал — хук оживає.
- **Хук виливає лише хвіст** (останні 6000 символів, обрізані по межі рядка). Журнал росте з
  кожною ітерацією; контекст рости не мусить.

⚠ У **інтерактивному** Claude Code новий хук вимагає підтвердження людини при першому запуску —
контриб'ютор, який відкриє репо руками, побачить один діалог. У headless `claude -p` (а саме так
працює луп) підтвердження не потрібне.

## Три жорсткі зупинки

Критерій «готово» тут — судження **моделі про себе**: вона сама створює файл `DONE` у корені.
Довіряти цьому без запобіжників не можна, тому цикл ніколи не крутиться нескінченно.

| # | Змінна | Дефолт | Що зупиняє |
|---|---|---|---|
| 1 | `MAX_ITER` | `6` | не більше N кругів |
| 2 | `K_FAILURES` | `3` | гейт червоний K разів **поспіль** |
| 3 | `NO_IMPROVEMENT` | `2` | метрика не зросла N ітерацій — плато |

Третя — найцікавіша. Перша каже «досить крутитись». Третя каже «крутитись більше **немає
сенсу**»: метрика вийшла на плато. Саме вона ловить агента, який уперся в питання, на яке не
має права відповісти сам, і чесно поставив задачі статус `blocked`.

`MAX_ITER` рахує зовнішні task-checkpoints, не всі звернення до моделей. Кожен checkpoint
запускає headless coordinator-а, а той послідовно делегує RED, GREEN і REVIEW трьом project
agents. Отже звичайна ітерація використовує чотири контексти; повторне виправлення після review
може використати більше.

```bash
MAX_ITER=3 npm run ralph -- --feature custom-alias
```

## Чому тут немає зупинки по бюджету

Вона була — і виявилась несправжньою. Вартість ходу приходить у події `result`, тобто вже
**після** того, як її витрачено: луп звіряв суму перед наступною ітерацією, тож перевитрата
дорівнювала вартості цілого ходу. Виміряно живим прогоном: `$1.93` при `BUDGET_USD=1`.

Стелею вона ніколи не була. `MAX_ITER` спрацьовує **до** нового coordinator-а й тому обмежує
кількість зовнішніх checkpoints, але не субагентні виклики всередині них. Реальну фінансову
стелю треба виставляти у провайдера.

Прибравши бюджет, ми прибрали і єдину причину, з якої ранер узагалі парсив `stream-json`. А з
нею — і всю таблицю адаптерів під кожен інструмент: без вартості всі агенти запускаються
однаково, і різниця між ними живе в одному рядку `AGENT_CMD`.

## Метрика — факт із диска, не думка

Модель може написати «зробив». Метрика цього не вміє: це кількість рядків `done` у трекері
**цільової** фічі, і нічого більше.

Трекери інших пакетів у метрику не входять свідомо. Інакше чужий прогрес — хоч би й від людини,
яка паралельно закриває інший пакет, — заховав би наше плато, і зупинка №3 не спрацювала б там,
де мусила.

## Гейт щоітерації

```
npm run lint && npm run test:fast && npm run links:check
```

Без `npm run test:e2e`: браузер довгий, і його ганяє сам агент там, де цього вимагає DoD
задачі. Гейт лупа — не заміна `npm run verify`, а швидка перевірка після кожного ходу.

⚠ `links:check` і `tools:check` обходять файлову систему, а не git, тож `.gitignore` їх не
стримує. `JOURNAL.md` вони пропускають окремим винятком: без нього агент, який чесно записав
«спіткнувся на `[[T2]]`», завалив би власний прогін, і причина була б цілком незрозуміла.

## Права агента

Права живуть у рядку `AGENT_CMD`, окремої змінної під них немає. Для Claude це
`--permission-mode auto`, і це не косметика, а умова, за якої луп узагалі щось робить.

У режимі `acceptEdits` агент може правити файли — і **не може виконати жодної команди**. У
headless підтвердити нікому, тож `node -e "console.log(1)"` віддає «This command requires
approval». Виміряно живим прогоном: агент чесно написав червоний тест, не зміг його запустити,
поставив задачі `blocked`, і ітерація коштувала `$1.93`, не давши ані коміта, ані зеленого.
`RED → GREEN → GATE → COMMIT` там нездійсненний за конструкцією.

В `auto` ті самі `node`, `npm run lint` і `git` проходять через класифікатор без підтвердження.
Тому ранер **відмовляється стартувати** (`exit 2`), якщо побачить `acceptEdits` будь-де в
`AGENT_CMD`: прогін, який не може виконати власний протокол, — це не прогін, а рахунок.

⚠ `permissions.allow` тут не рятує: воркспейс, для якого не прийнято trust-діалог, цей ключ
ігнорує, а свіжий клон завжди недовірений.

## Гілка й стеля автономії

**Гілку створюєш ти:** `git checkout -b feat/<slug>`. Ранер не пише в git жодного разу — він
лише **відмовляється стартувати** (`exit 2`), якщо HEAD стоїть на `main`.

Це слабша конструкція, ніж «скрипт створює гілку сам», і слабша свідомо. Ранер, який робить
`checkout -b`, мусить уміти писати в git; ранер, який просто відмовляється, не мусить нічого.
Стеля автономії тепер тримається на **відмові**, а не на дії.

Стеля лишається та сама — **коміт у локальну гілку**. `git push` і `git checkout main`
заборонені промптом. Найгірше, що може статися за ніч, — смітна локальна гілка, яку видаляють
одним рядком.

Брудне робоче дерево справжній прогін відмовляється чіпати: `exit 2`, доки не передано
`--allow-dirty`. Перевірка `main` від `--allow-dirty` **не залежить**: обійти її можна лише
руками, створивши гілку. Сухий прогін дерево не чіпає й тому на брудність не зважає взагалі.

## Агент-агностика

Промпт іде **останнім аргументом**, `stdio: 'inherit'`, читається код виходу. Це весь інтерфейс:
жодного `if` під конкретний інструмент, жодних адаптерів. Розкоментуй свій рядок у `ralph.mjs`
або передай його змінною:

| Агент | Рядок |
|---|---|
| Claude Code (дефолт) | `claude -p --permission-mode auto` |
| Codex | `codex exec --sandbox workspace-write` |
| GitHub Copilot | `copilot --allow-all-tools -s -p` |
| Cursor | `cursor-agent --print --force` |

```bash
AGENT_CMD='codex exec --sandbox workspace-write' npm run ralph -- --feature qr-codes
```

⚠ **Порядок прапорців — не косметика.** Промпт додається останнім, тож у `copilot` він стає
**значенням** `-p`, і рядок мусить закінчуватись саме на `-p`. Той самий `-p` означає різне
всюди: у `claude` це «headless», у `copilot` — «промпт», у `cursor-agent` — «друкуй у консоль»,
а в `codex exec` це взагалі `--profile`, і промпт там позиційний.

Прапорці одного агента іншому не передавай: `--permission-mode` — прапорець Claude Code, і
`codex exec` падає на ньому з «unexpected argument».

Код виходу чужого агента читається обов'язково. Агент, який навіть не стартував, інакше
виглядав би як успішний хід, і луп крутив би порожні кола, доки не спрацює зупинка №3.

⚠ Промпт **не** каже `/implement`. У headless `claude -p` слеш-команди не спавняться, тому він
посилається на `SKILL.md` шляхом і прямо вимагає послідовний dispatch `test-author → implementer
→ reviewer`. Для іншого `AGENT_CMD` потрібні еквівалентні іменовані ролі; тихого single-agent
fallback немає.

⚠ Хук пам'яті — механізм Claude Code. Інший агент його не запустить, і журнал йому ніхто не
покаже: `JOURNAL.md` тоді просто накопичується, а читає його людина.

## Коди виходу

| Код | Значення |
|---|---|
| `0` | модель створила `DONE` (або `--dry-run` відпрацював) |
| `1` | спрацювала одна з трьох жорстких зупинок |
| `2` | зламана конфігурація: немає `PROMPT.md`, лежить старий `DONE`, невідомий slug, не передано `--feature`, режим `acceptEdits`, HEAD на `main`, брудне дерево |
| `3` | у трекері цілі не лишилось жодного `todo` — робити нема чого |

## Чому `--dry-run` стоїть у воротах

`npm run verify` ганяє `npm run ralph -- --dry-run` як звичайні ворота. Це доводить за нуль
токенів, що ранер стартує, бачить промпт і знає свій беклог. Ранер, який ламається лише тоді,
коли його запускають по-справжньому, — це не ранер, а обіцянка.

Саме тому димовий прогін не вимагає `--feature`. Ворота, у які зашито ім'я пакета, почервоніли б
того дня, коли той пакет заshipиться, — а репо, де все зроблено, не є зламаним репо.
