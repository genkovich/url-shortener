## Why (user story)

As a **visitor**, I want to be told that I hit a limit and how long to wait, so that I stop clicking and come back once instead of ten times.

Spec US-02. AC-02 (the server sends a `Retry-After` the browser can read), AC-07 (the form shows the limit and the wait).

Today the browser already renders the server's `error` string verbatim, so a `429` displays the words **`rate limited`** — English, and silent about the wait. That is the red state this task starts from.

    $('error').textContent = Number.isInteger(seconds) && seconds > 0

## API contract

_API surface: none — the frontend is a client of `POST /api/shorten`._

The page and the API are same-origin, so `res.headers.get('Retry-After')` is readable without ceremony. A deployment that ever serves the frontend from another origin would need `Access-Control-Expose-Headers: Retry-After`, because `Retry-After` is not one of the CORS-safelisted response headers. Not a concern today; worth one line here so it is not a mystery later.

## Acceptance criteria (GWT)

- [ ] **AC-t4-1 (the wait is shown — AC-07):** Given the server answers `429` with `Retry-After: 3`, when the form is submitted, then `#error` reads `Забагато запитів. Спробуй за 3 с.`
- [ ] **AC-t4-2 (missing header — AC-07):** Given a `429` with no `Retry-After`, then `#error` reads `Забагато запитів. Спробуй трохи пізніше.` — the sentence still tells the visitor what happened. `res.headers.get()` returns `null` and `Number(null)` is `0`, so the same predicate that guards the parse also catches the absence.
- [ ] **AC-t4-3 (the visitor's input survives — AC-07):** After a `429`, `#url` still contains what the visitor typed. Only a `201` clears it.
- [ ] **AC-t4-4 (a limit is not a mistake):** The `429` message is rendered in `--warn`, not in the red `--error` used for `400`. A `400` says *you typed something wrong*; a `429` says *you did nothing wrong, wait a second*. Painting them the same colour tells the visitor to fix a URL that is perfectly fine.
- [ ] **AC-t4-5 (no stale success):** Given a previous submission succeeded and `#result` is visible, when the next one is refused, then `#result` is hidden. Otherwise the page shows a short URL and an error at the same time, and the short URL is the one it invites you to copy.
- [ ] **AC-t4-6 (other errors unchanged):** `400`, `404` and `501` render exactly as before, in `--error`, and `501` keeps its `Ще не реалізовано (фіча: …)` wording.
- [ ] **AC-t4-7 (no rule in the UI):** `grep -nE "RATE_LIMIT|60000|token|bucket|setInterval" src/public/app.js` returns nothing. The browser knows the number of seconds it was handed, and nothing else.

## Edge cases

| Case | Behaviour |
|---|---|
| `Retry-After` absent | `res.headers.get('Retry-After')` returns `null`; `Number(null)` is `0`, which fails `seconds > 0`, which selects the sentence without a number. Never render `Спробуй за 0 с.` — the visitor would click again immediately and be refused again, which is exactly the busy-wait the server-side `Math.max(1, …)` exists to prevent. The browser must not be the second place that bug can appear. |
| `Retry-After: NaN` on the wire | Possible if a bad `RATE_LIMIT_WINDOW_MS` slipped past T3 (measured: Express sends the header `NaN` without complaint). `Number('NaN')` is `NaN`, `Number.isInteger(NaN)` is `false`, so the fallback sentence appears. The frontend degrades; it does not print `NaN` at the visitor. |
| A countdown that ticks down | Tempting, and out of scope. It needs a timer in the page, a re-render, and a decision about what to show at zero. The static number is honest: it is what the server said at the moment it refused. |
| Duplicating the limit in the browser | Would let the page pre-empt the server — refusing the visitor before the request, using a capacity the operator may have changed an hour ago. Every rule is server-side (`sad.md` §5). AC-t4-7 pins it. |
| Stale `#result` beside an error | Pre-existing: nothing in the current `if (!res.ok)` branch hides it. A `400` used to leave the last short URL on screen too. The `429` makes it obvious, because a refused visitor is precisely the one who will click *copy*. Fixed here for every status, not only for `429`. |
| Amber for `429`, red for `400` | Not decoration. `custom-alias` made the same distinction between `400` and `409` in status codes; this is that distinction made visible. The visitor's next action differs: fix the input, or wait. |
| Cross-origin frontend | `Retry-After` is not CORS-safelisted, so `headers.get` would return `null` and the page would fall back to the sentence without a number. Same-origin here, so it does not arise; the fallback means it would not break if it ever did. |
