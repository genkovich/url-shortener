# Технічне завдання: сторінка підготовки до роботи з `url-shortener`

> Цей файл призначений для реалізації **окремої сторінки на сайті
> `agenticengineering.it.com`**, а не для додавання сторінки до самого застосунку
> `url-shortener`.

## Результат

Створити україномовну сторінку підготовки за матеріалами
[`README.md`](../README.md), подібну за форматом до
[`/preparework`](https://agenticengineering.it.com/preparework): темна одноколонкова
сторінка, послідовні кроки, команди з кнопками копіювання та інтерактивний чекліст.

Рекомендована адреса:

```text
https://agenticengineering.it.com/prepare-url-shortener
```

Назву маршруту можна змінити перед реалізацією. Важливо, щоб сторінка мала окрему
постійну адресу й відкривалася напряму після перезавантаження браузера.

## Джерело правди

- Зміст і команди беруться з [`README.md`](../README.md).
- Правила репозиторію беруться з [`AGENTS.md`](../AGENTS.md).
- Архітектурні подробиці не треба переносити на сторінку повністю: за потреби дати
  посилання на [`architecture-map.md`](./architecture-map.md).
- Сторінка пояснює підготовку до практики, але не замінює README.
- Не вигадувати нові команди, залежності, плагіни або обов’язкові сервіси.

У компоненті сторінки варто залишити коментар із посиланням на README, щоб під час
наступного оновлення було зрозуміло, звідки синхронізувати текст.

## Що не треба робити

- Не додавати цю сторінку до `src/public/` репозиторію `url-shortener`.
- Не змінювати Express, SQLite, API або поточну сторінку скорочення посилань.
- Не намагатися виконувати команди на сервері від імені відвідувача.
- Не збирати токени, паролі, вміст термінала або інші локальні дані користувача.
- Не завантажувати README через мережу під час кожного відкриття сторінки.
- Не копіювати зібрані JavaScript/CSS-файли сторінки-референсу. Треба використати її
  наявні вихідні компоненти та стилі в кодовій базі сайту.

## Структура сторінки

### 1. Вступний блок

Малий технічний підпис:

```text
// preparation
```

Заголовок:

```text
Підготовка до практики з url-shortener
```

Короткий опис:

```text
Встанови інструменти, перевір середовище та запусти проєкт до початку практики.
Плагіни, Docker і окремий SQLite не потрібні.
```

Бейджі під заголовком:

- `20–40 хвилин`;
- `macOS · Linux · Windows`;
- `Node.js 24 LTS`;
- `без зовнішніх плагінів`.

Інформаційна картка:

```text
Пройди всі кроки заздалегідь. У кінці npm run verify має завершитися без помилок,
а сторінка застосунку — відкритися на localhost:3000.
```

Час указаний орієнтовно: завантаження залежностей і Chromium залежить від швидкості
інтернету та комп’ютера.

### 2. Що знадобиться

Показати чотири компактні картки:

1. **Git** — для клонування репозиторію та роботи з гілкою.
2. **Node.js 24 LTS + npm** — npm уже постачається разом із Node.js.
3. **Один coding agent** — Claude Code, Codex CLI, GitHub Copilot CLI або Cursor Agent.
4. **Інтернет** — для `npm ci`, Chromium і роботи агента.

Під картками окремо зазначити:

```text
Docker, окремий SQLite, глобальні npm-пакети та SDD-плагіни не потрібні.
```

### 3. Встановлення інструментів за операційною системою

Додати перемикач із трьома вкладками:

- `macOS`;
- `Linux`;
- `Windows`.

Одночасно показувати лише одну вкладку. Обрану операційну систему можна запам’ятати
в `localStorage`, але це не обов’язково для першої версії.

#### macOS

Послідовність:

1. Встановити Homebrew офіційним install script.
2. Виконати надруковані Homebrew команди `Next steps`, щоб додати `brew` до `PATH`.
3. Встановити Git через Homebrew.
4. Встановити `nvm`.
5. Перезапустити Terminal.
6. Встановити Node.js 24 LTS і зробити його типовим.

Команди:

```bash
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
```

Після цього користувач виконує показані інсталятором команди `Next steps`, а тоді:

```bash
brew --version
brew install git
git --version
```

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

Якщо Homebrew покаже системний діалог встановлення Command Line Tools, користувач має
погодитися; окремої ручної команди для них у підготовці немає.

```bash
command -v nvm
nvm install 24
nvm alias default 24
```

#### Linux

Спочатку показати блоки пакетних менеджерів. Пояснити, що треба виконати лише один
варіант для свого дистрибутива.

```bash
# Debian / Ubuntu
sudo apt update
sudo apt install -y git curl

# Fedora
sudo dnf install -y git curl

# Arch Linux
sudo pacman -S --needed git curl
```

Потім установити `nvm` і Node.js 24 LTS:

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

```bash
command -v nvm
nvm install 24
nvm alias default 24
```

Перед останнім блоком пояснити, що термінал треба перезапустити або виконати
`source ~/.bashrc` чи `source ~/.zshrc`.

#### Windows

Показати PowerShell-команду для Git:

```powershell
winget install --id Git.Git -e --source winget
```

Далі дати нумеровану інструкцію:

1. Відкрити архів останньої версії Node.js 24 LTS.
2. Завантажити `.msi` для `x64` або `arm64`.
3. Встановити з типовими налаштуваннями.
4. Перезапустити PowerShell.

Додати кнопки на офіційні сторінки Git і Node.js 24, які вже вказані в README.

Наприкінці кожної вкладки показати перевірку:

```bash
git --version
node --version
npm --version
```

Окремим блоком зазначити: достатньо, щоб версію показував хоча б один агент.

```bash
claude --version
codex --version
copilot --version
cursor-agent --version
```

### 4. Клонування та встановлення проєкту

Крок 1 — клонування:

```bash
git clone https://github.com/genkovich/url-shortener.git
cd url-shortener
```

Крок 2 — лише для macOS/Linux:

```bash
nvm use
```

Крок 3 — залежності та Chromium.

Для macOS/Windows:

```bash
npm ci
npx playwright install chromium
```

Для Linux:

```bash
npm ci
npx playwright install --with-deps chromium
```

У кожному блоці команд має бути кнопка `Копіювати`. Після натискання вона на короткий
час змінює текст на `Скопійовано`.

### 5. Перевірка середовища

Показати один блок:

```bash
npm run doctor
npm run verify
```

Пояснення під ним:

- `doctor` перевіряє Node.js, npm, Git, coding agents і `node_modules`;
- `verify` запускає всі детерміновані ворота, включно з браузерним E2E-тестом;
- зелений `verify` означає, що середовище готове до практики;
- якщо команда червона, користувач має виправити першу показану помилку й повторити
  `npm run verify`.

Не треба імітувати результат цих команд на вебсторінці: вони виконуються локально в
терміналі користувача.

### 6. Запуск застосунку

Команда:

```bash
npm run dev
```

Після команди показати велику кнопку:

```text
Відкрити localhost:3000 →
```

Кнопка веде на `http://localhost:3000` і відкривається в новій вкладці. Треба прямо
пояснити, що вона спрацює лише тоді, коли локальний сервер уже запущено.

Додатковий текст:

```text
Під час першого запуску застосунок автоматично створить локальну базу
data/links.db. Окремо встановлювати або налаштовувати SQLite не потрібно.
```

### 7. Що відбуватиметься під час практики

Подати як вертикальну послідовність:

1. Учасник читає `AGENTS.md` та `docs/architecture-map.md`.
2. Обирає фічу з `docs/roadmap.md`.
3. Відкриває специфікацію й поточну задачу в `docs/features/<slug>/`.
4. Запускає вендорений скіл `implement` у своєму інструменті.
5. Скіл веде задачу циклом `RED → GREEN → REFACTOR → GATE → COMMIT`.
6. Після кожної правки запускається `npm run verify`.
7. Людина переглядає результат; `git push` виконується лише окремим свідомим рішенням.

Поруч показати команди запуску скіла:

| Інструмент | Як запустити |
|---|---|
| Claude Code | `/implement <slug>` |
| Codex | `$sdd-implement <slug>` |
| Cursor | `/` → `sdd-implement` |

Окремий акцент:

```text
Нічого встановлювати як SDD-плагін не треба: потрібний скіл уже лежить у репозиторії.
```

### 8. Довідник команд

Таблицю команд із README показати в акордеоні `Дивитися всі команди`, щоб вона не
перевантажувала основний сценарій. Усередині мають бути:

- `npm run dev`;
- `npm run lint`;
- `npm run test:unit`;
- `npm run test:integration`;
- `npm run test:fast`;
- `npm run test:e2e`;
- `npm test`;
- `npm run gate`;
- `npm run doctor`;
- `npm run verify`;
- `npm run tools:check`;
- `npm run links:check`.

### 9. Автономний луп

Зробити окремий необов’язковий блок із позначкою `Для самостійної практики`:

```bash
git checkout -b feat/<slug>
npm run ralph -- --feature <slug> --dry-run
npm run ralph -- --feature <slug>
```

Пояснити дві межі:

- луп не працює на `main`;
- луп не виконує `git push`.

Додати посилання на [`loop/README.md`](../loop/README.md), але не переносити весь його
вміст на сторінку підготовки.

### 10. Фінальний чекліст

Пункти:

1. Git показує версію.
2. Node.js має основну версію `24`.
3. npm показує версію.
4. Хоча б один coding agent установлено й авторизовано.
5. Репозиторій склоновано.
6. `npm ci` завершився успішно.
7. Chromium для Playwright установлено.
8. `npm run doctor` не показує блокувальних проблем.
9. `npm run verify` завершується зеленим результатом.
10. `http://localhost:3000` відкриває сторінку скорочувача.

Над списком показувати:

- `N з 10 виконано`;
- відсоток;
- смугу прогресу.

Стан чекліста зберігати лише в браузері:

```text
localStorage: url-shortener.prepare.v1
```

Не надсилати цей стан на сервер. Додати непомітну кнопку `Скинути прогрес`.

## Що відбуватиметься на практиці

### Для відвідувача

1. Людина відкриває публічну сторінку.
2. Обирає свою операційну систему.
3. Копіює команди по одній і виконує їх у локальному терміналі.
4. Після кожного успішного кроку ставить галочку.
5. Браузер запам’ятовує галочки лише на цьому пристрої.
6. Після `npm run verify` людина запускає `npm run dev` і відкриває локальний застосунок.

Вебсторінка нічого не встановлює сама. Вона лише веде користувача правильним порядком,
дає команди для копіювання й допомагає не пропустити крок.

### Для сайту

1. У кодовій базі `agenticengineering.it.com` з’являється новий компонент сторінки.
2. У маршрутизаторі реєструється `/prepare-url-shortener`.
3. Під час збірки сторінка потрапляє до того самого статичного пакета, що й
   `/preparework`.
4. Після деплою Cloudflare віддає головний HTML, а клієнтський маршрутизатор показує
   новий компонент.
5. Жодного нового API або сховища даних не потрібно; чекліст працює через
   `localStorage`.

### Для підтримки змісту

README залишається канонічним джерелом. Коли в README змінюється версія Node.js, команда
встановлення або перелік воріт, текст сторінки треба оновити в тому самому pull request.
Автоматичне завантаження README під час відкриття сторінки не потрібне: воно зробить
сторінку залежною від GitHub і погіршить контроль над версткою.

## Візуальні вимоги

Використати стиль і готові компоненти `/preparework`, але не копіювати зібрані assets:

- темне тло;
- один центральний контейнер приблизно `720–768px`;
- білий основний текст і приглушений другорядний;
- синій акцент для номерів, посилань і прогресу;
- технічні мітки моноширинним шрифтом;
- картки з тонкою рамкою;
- окремі темні блоки команд;
- достатні відступи між секціями;
- адаптивність від `320px`;
- горизонтальне прокручування лише всередині довгих блоків команд або таблиць.

На мобільному екрані вкладки ОС можуть переноситися на наступний рядок. Кнопка
копіювання має лишатися видимою, а текст команд — не ламати ширину сторінки.

## Доступність

- Один `h1`, далі логічна ієрархія `h2`/`h3`.
- Вкладки ОС доступні з клавіатури й мають коректні ARIA-ролі.
- Чекліст реалізовано справжніми `input type="checkbox"` або еквівалентними доступними
  контролами.
- Для кнопки копіювання є текстовий стан успіху.
- Видимий `focus` не прибирається.
- Контраст тексту й фону відповідає WCAG AA.
- Анімації не є необхідними для розуміння сторінки та враховують
  `prefers-reduced-motion`.

`/preparework` і повторити прийняту там організацію, а не створювати другу систему стилів
або маршрутизації.

## Критерії готовності

- Сторінка повністю українською, крім назв інструментів і команд.
- Усі інструкції відповідають поточному README.
- Операційні системи розділені й не змушують читати зайве.
- Кожен блок команд копіюється однією кнопкою.
- Чекліст зберігається після перезавантаження та скидається вручну.
- Жодні персональні або локальні дані не надсилаються на сервер.
- Пряме відкриття `/prepare-url-shortener` не повертає `404`.
- Зовнішні посилання відкриваються без втрати прогресу сторінки.
- На мобільному немає горизонтального прокручування всієї сторінки.
- Сторінка має title, description, canonical та Open Graph metadata.
- Перевірені команди для macOS, Linux і Windows.
- У footer є посилання на репозиторій і на README як канонічну інструкцію.
