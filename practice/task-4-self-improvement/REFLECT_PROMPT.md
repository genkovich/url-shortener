Ти виконуєш одну reflection-ітерацію над housekeeping policy цього репозиторію.

## Межі

- Прочитай `AGENTS.md`, `practice/task-1-housekeeping/PROMPT.md` і надані записи journal.
- Працюй тільки з повторюваною проблемою, яку прямо підтверджують останні два записи.
- Можна змінити рівно два файли:
  - `practice/task-1-housekeeping/PROMPT.md`;
  - `practice/task-4-self-improvement/eval.mjs`.
- Не змінюй product-код, runner, тести продукту, SDD-пакети або цей reflection prompt.
- Не перемикай гілку, не виконуй `git push` і не створюй PR.

## Контракт eval

`eval.mjs` має бути детермінованим, не потребувати npm-залежностей і підтримувати два режими:

- без аргументів — прочитати поточний `practice/task-1-housekeeping/PROMPT.md` із диска;
- `--ref <git-ref>` — прочитати цей самий файл через `git show <git-ref>:<path>`.

Eval завершується кодом `0`, якщо правило присутнє, кодом `1`, якщо його немає, і кодом `2`
для неправильних аргументів або неможливого Git-ref. Вивід має називати перевірене правило й
показувати `GREEN` або `RED`.

## Протокол

1. Перетвори повторюваний failure на одне загальне правило, а не на опис конкретного інциденту.
2. Спочатку створи або посиль `eval.mjs`; housekeeping prompt поки не змінюй.
3. Запусти `node practice/task-4-self-improvement/eval.mjs --ref HEAD` і отримай RED з кодом `1`.
4. Додай мінімальне правило в `practice/task-1-housekeeping/PROMPT.md`.
5. Повтори eval без `--ref` і отримай GREEN з кодом `0`. Потім запусти `npm run lint`.
6. Переглянь diff: лише два дозволені файли, без послаблення наявних правил.
7. Створи рівно один локальний коміт `chore(self-improvement): <яке правило додано>`.

Якщо evidence недостатньо, RED отримати не вдалося або правило вже існує — нічого не змінюй,
не коміть і заверши відповіддю `NO_REFLECTION`.
