# Задача 2 — Ralph loop: запусти реалізацію фічі й подивись, як він працює

Це друга задача воркшопу. Репозиторій уже налаштований, залежності встановлені, а результат
першої задачі закомічений у її гілці. Тут ми нічого не програмуємо руками: створюємо окрему
чисту гілку, запускаємо готовий Ralph loop і спостерігаємо, як він реалізує фічу
`input-validation`.

Мета вправи — побачити три речі:

1. що саме відбувається під час кожної ітерації;
2. звідки агент отримує промпт і пам'ять про попередній хід;
3. чому прогрес вимірюється фактами з репозиторію, а не словами моделі.

Справжній запуск використовує AI-модель і може коштувати грошей. Не запускай його, доки
ведучий воркшопу не дозволив витрати. Команда нижче обмежує цикл двома ітераціями.

## Крок 1 — перейди в окрему чисту гілку

Переконайся, що результат першої задачі закомічений:

```bash
git status --short
```

Команда не має нічого надрукувати. Порожня відповідь означає, що робоче дерево чисте.

Створи нову гілку `feat/input-validation` від чистої `main` і відразу перейди в неї:

```bash
git switch -c feat/input-validation main
```

Ця команда не видаляє гілку першої задачі. Вона залишається на місці разом з усіма її
комітами.

Перевір результат:

```bash
git branch --show-current
git status --short
```

Перша команда має надрукувати:

```text
feat/input-validation
```

Друга знову не має нічого надрукувати. Якщо git каже, що гілка вже існує, або показує
незакомічені файли, зупинись і поклич ведучого. Не додавай до Ralph прапорець `--allow-dirty`:
він вимикає запобіжник від змішування чужих змін із роботою агента.

## Крок 2 — зафіксуй зелену стартову точку

```bash
npm run verify
```

Наприкінці має бути:

```text
verify OK — 6 ✓
```

Тепер ми знаємо, що репозиторій був справний до першої зміни Ralph.

## Крок 3 — подивись план без запуску агента

```bash
npm run ralph -- --feature input-validation --dry-run
```

`--dry-run` нічого не редагує, не викликає модель і не витрачає токени. Він лише збирає план
майбутнього запуску з файлів репозиторію.

Знайди у виводі такі рядки:

```text
ціль:             input-validation (todo: 5, done: 0)
гілка:            feat/input-validation (HEAD уже на «feat/input-validation» — лишиться як є)
гейт щоітерації:  npm run lint && npm run test:fast && npm run links:check
Метрика прогресу зараз: input-validation: 0 done, 5 todo.
```

Нижче Ralph покаже чотири запобіжники:

| Запобіжник | Що він обмежує |
|---|---|
| `MAX_ITER` | максимальну кількість ходів агента |
| `BUDGET_USD` | витрати перед стартом наступного ходу |
| `K_FAILURES` | кількість червоних гейтів поспіль |
| `NO_IMPROVEMENT` | кількість ходів, у яких не зросло `done` |

Наприкінці має бути `Жодного токена не витрачено`. Якщо `todo` і `done` відрізняються від
прикладу, ця гілка вже містить частину реалізації — покажи сухий прогін ведучому.

## Крок 4 — звідки береться промпт

Перед справжнім запуском відкрий статичний промпт одного ходу:

```bash
cat loop/PROMPT.md
```

Це не прихований системний текст. Промпт лежить у репозиторії, проходить code review і
змінюється звичайним комітом. У ньому записано протокол одного ходу:

```text
RED → GREEN → REFACTOR → GATE → COMMIT
```

Тобто агент має:

1. спочатку написати тест, який падає;
2. додати найменшу реалізацію, що робить тест зеленим;
3. прибрати зайву складність;
4. запустити гейт;
5. оновити трекер і зробити один локальний коміт на задачу.

Ralph збирає контекст ітерації з кількох частин:

```text
loop/PROMPT.md
        +
динамічна ціль: номер ітерації, slug, гілка та шляхи до SDD-пакета
        +
факти з SessionStart-хука: git, tracker.md і хвіст JOURNAL.md
        ↓
новий процес claude -p
        ↓
AGENTS.md → architecture-map.md → пакет input-validation
```

### Статична частина

[loop/PROMPT.md](../../loop/PROMPT.md) однаковий у кожній ітерації. Він задає порядок читання,
TDD-протокол, заборону `git push`, правила блокування і умову створення `DONE`.

### Динамічна ціль

[loop/ralph.mjs](../../loop/ralph.mjs) щоразу дописує до промпту номер ітерації, slug
`input-validation`, назву гілки та точні шляхи до специфікації й трекера. Тому один і той самий
рушій може запускати різні SDD-пакети без окремого промпту для кожної фічі.

### Факти й пам'ять

На старті нової Claude-сесії хук
[.claude/hooks/loop-memory.mjs](../../.claude/hooks/loop-memory.mjs) сам читає поточну гілку,
чистоту дерева, останні коміти та статуси задач. Після фактів він додає хвіст
`loop/JOURNAL.md` — короткий самозвіт попередніх ітерацій.

Джерела навмисно розділені:

- git і `tracker.md` — факти з диска, їм віримо;
- `JOURNAL.md` — пояснення моделі, його звіряємо з фактами.

## Крок 5 — запусти реалізацію

Покажи ведучому сухий прогін. Після дозволу скопіюй весь рядок як одну команду:

```bash
MAX_ITER=2 BUDGET_USD=4 K_FAILURES=2 NO_IMPROVEMENT=2 npm run ralph -- --feature input-validation
```

У цій вправі Ralph зробить не більше двох ходів. `BUDGET_USD=4` перевіряється перед наступною
ітерацією, тому це зупинка, а не точна стеля: вже запущений хід завершується й може перевищити
залишок бюджету.

На початку буде видно:

```text
ralph: ціль input-validation (todo: 5, done: 0)
гілка:            feat/input-validation (уже не main — лишаємо як є)
--- ітерація 1/2  (витрачено $0.0000 з $4) ---
```

Поки команда працює, нічого не вводь у цей термінал. Щоб аварійно зупинити саме Ralph, натисни
`Ctrl+C` один раз і дочекайся повідомлення `Перервано на ітерації ...`.

## Крок 6 — як дивитися за виконанням

### Заголовок ітерації

```text
--- ітерація 2/2  (витрачено $1.2345 з $4) ---
```

Він показує номер поточного ходу й вартість **завершених** ходів цього запуску. Вартість
поточного ходу Ralph дізнається лише після його завершення.

### Повідомлення агента

Між заголовком і підсумком Ralph наживо друкує текстові повідомлення агента: яку задачу він
вибрав, який тест пише, що змінив і які перевірки запускає. Це корисний live-коментар, але не
доказ виконання. Докази з'являться в трекері, git і гейті.

### Результат гейта

Гейт кожної ітерації:

```text
npm run lint && npm run test:fast && npm run links:check
```

Зелений гейт не друкує окремої довгої таблиці — цикл просто переходить до метрики. Червоний
гейт видно явно:

```text
гейт червоний (1/2 поспіль)
```

Після двох таких результатів Ralph зупиниться.

### Метрика прогресу

Після успішно закритої задачі буде:

```text
прогрес: input-validation: 1 done, 4 todo
```

Якщо агент завершив хід, але кількість `done` не зросла:

```text
прогресу немає (1/2): input-validation: 0 done, 5 todo
```

Метрика не аналізує красиві відповіді моделі. Ralph рахує рядки зі статусом `done` безпосередньо
у `docs/features/input-validation/tasks/tracker.md`.

### Причина зупинки

Останній рядок має форму:

```text
=== СТОП ...: ліміт ітерацій (MAX_ITER=2) вичерпано без DONE (ітерацій: 2, витрачено: $...) ===
```

Код завершення `1` тут часто означає штатну роботу запобіжника, а не падіння програми. Дві
ітерації закінчилися, але з п'яти задач фічі ще залишилися невиконані.

## Крок 7 — де подивитися результат після запуску

### Реальний стан задач

```bash
cat docs/features/input-validation/tasks/tracker.md
```

Колонка `Status` показує `todo`, `blocked` або `done`. Саме з цього файла Ralph бере метрику.

### Локальні коміти

```bash
git log --oneline --decorate -6
```

Нові коміти агента будуть зверху. За протоколом одна завершена задача має дати один коміт.
Ralph не виконує `git push`, тому ці коміти залишаються тільки на твоєму комп'ютері.

### Витрати, прогрес і причина зупинки

```bash
tail -n 30 cost.log
```

`cost.log` — аудит запуску. Після кожної завершеної ітерації він отримує рядок `iter=...`, ціну
цього ходу й накопичений `total`. Туди ж записуються зміна метрики та фінальна причина зупинки.

### Що має знати наступна ітерація

```bash
tail -n 40 loop/JOURNAL.md
```

Наприкінці ходу агент дописує сюди три короткі нотатки: що зробив, де спіткнувся і що варто
знати наступному ходу. Файл не комітиться й не є метрикою прогресу.

## Крок 8 — фінальна перевірка

Прибери службовий сигнал завершення, якщо агент устиг створити його. Команда не чіпає код або
коміти:

```bash
node -e "require('fs').rmSync('DONE',{force:true})"
```

Запусти всі детерміновані ворота:

```bash
npm run verify
```

Очікуваний результат:

```text
verify OK — 6 ✓
```

На цьому вправа закінчена. Не запускай цикл повторно без ведучого: кожен новий запуск починає
власний лічильник бюджету з нуля, хоча попередня історія залишається в `cost.log`.

# Чому сліпий цикл працює саме так

Кожна ітерація — новий процес `claude -p` із чистим контекстом. Модель не продовжує довгий чат
і не покладається на власну пам'ять. Усе, що має пережити хід, мусить бути записане на диск.

Це дає чітку ієрархію довіри:

1. **Git** доводить, які зміни реально закомічені.
2. **Трекер** дає монотонну метрику `done` для цільової фічі.
3. **Гейт** незалежно перевіряє код, тести й документацію.
4. **Журнал** передає корисний досвід, але лишається самозвітом моделі.
5. **Жорсткі зупинки** не дозволяють нескінченно витрачати час і гроші без прогресу.

Довгий діалог може накопичувати застарілі припущення. Холодний старт змушує кожну ітерацію
заново звірятися з актуальним git, трекером і специфікацією. Ціна цього підходу — дисципліна:
стан, якого немає на диску, для наступної ітерації не існує.

Повний технічний опис лежить у [loop/README.md](../../loop/README.md). Специфікація цільової
фічі — у [docs/features/input-validation/spec.md](../../docs/features/input-validation/spec.md),
а її задачі — у
[docs/features/input-validation/tasks/tracker.md](../../docs/features/input-validation/tasks/tracker.md).
