# Задача 2 — Ralph loop: запусти й простеж два ходи

У задачі 1 ми самі зібрали маленький housekeeping-цикл. Тепер репозиторій уже налаштований,
залежності встановлені, а результат першої задачі закомічений у `chore/housekeeping`.

У цій вправі ми не пишемо код руками. Перейдемо в чисту гілку, запустимо готовий Ralph loop над
фічею `input-validation` і простежимо максимум **два ходи з п'яти**. Мета — не завершити всю
фічу, а побачити, звідки агент бере завдання, як наступний хід дізнається про попередній і чому
результат визначає репозиторій, а не звіт моделі.

> Справжній запуск викликає AI-модель і може коштувати грошей. У Ralph немає грошового ліміту:
> `MAX_ITER=2` обмежує два зовнішні checkpoints, але кожен із них викликає coordinator і трьох
> субагентів. Фінансову межу ведучий виставляє у провайдера. Запускай модель тільки після дозволу.

У задачі є два альтернативні маршрути:

- **Ralph** — основний маршрут нижче: новий headless-процес на кожен хід;
- **Claude Code** — [готовий implementation-промпт](./GOAL_PROMPT.md), який виконує ту саму
  фічу без `ralph.mjs`.

Обери один маршрут. Не запускай Ralph і Claude Code одночасно в одній гілці.

## Крок 1 — перейди в чисту гілку

Переконайся, що задача 1 закомічена:

```bash
git status --short
```

Порожня відповідь означає, що все чисто. Тепер створи нову гілку від `main`:

```bash
git switch -c feat/input-validation main
```

Гілка `chore/housekeeping` не зникає — разом із задачею 1 вона лишається в історії git.

Перевір нову стартову точку:

```bash
git branch --show-current
git status --short
```

Має бути видно `feat/input-validation`, а після нього — порожня відповідь. Якщо гілка вже існує
або git показує незакомічені файли, зупинись і поклич ведучого. Не використовуй
`--allow-dirty`: він вимикає захист від змішування змін.

## Крок 2 — сухий прогін

```bash
MAX_ITER=2 K_FAILURES=2 NO_IMPROVEMENT=2 npm run ralph -- --feature input-validation --dry-run
```

Сухий прогін не викликає модель і не редагує файли. Він має показати:

```text
агент:    claude -p --permission-mode auto
промпт:   .../loop/PROMPT.md
ціль:     input-validation: 0 done, 5 todo
трекер:   .../docs/features/input-validation/tasks/tracker.md
гілка:    feat/input-validation · дерево чисте
старт:    готовий
зупинки:  MAX_ITER=2 · K_FAILURES=2 · NO_IMPROVEMENT=2
Жодного токена не витрачено.
```

Не переходь далі, якщо `старт` не `готовий` або початкова метрика не `0 done, 5 todo`.

## Крок 3 — що отримає агент

Ralph не вигадує завдання. Один хід складається з трьох частин:

```text
loop/PROMPT.md
  + номер ітерації, slug, гілка та шляхи до SDD-пакета
  + свіжі факти з git і трекера та журнал попередніх ходів
  ↓
нова headless-сесія Claude coordinator
  → test-author → implementer → reviewer
```

Подивись лише початок статичного промпту:

```bash
sed -n '1,52p' loop/PROMPT.md
```

Там видно головний протокол і розподіл ролей:

```text
test-author: RED
  → implementer: GREEN → REFACTOR → GATE
  → reviewer: REVIEW_CLEAN
  → coordinator: tracker → done, COMMIT
```

Coordinator бере першу доступну `todo`-задачу з трекера та послідовно делегує три ізольовані
ролі. Сам він не пише тест або продукт-код; після `REVIEW_CLEAN` оновлює tracker і комітить одну
задачу.

Щоразу [ralph.mjs](../../loop/ralph.mjs) дописує до цього промпту номер ітерації та точні шляхи.
На старті сесії [loop-memory.mjs](../../.claude/hooks/loop-memory.mjs) додає факти з git і
трекера, а також хвіст `loop/JOURNAL.md`. Тому наступна сесія має свіжий контекст, хоча не
пам'ятає попередній чат.

## Крок 4 — на що дивитися під час запуску

Спочатку Ralph надрукує стабільний заголовок:

```text
ralph: ціль input-validation (todo: 5, done: 0), гілка feat/input-validation
--- ітерація 1/2 ---
```

Посередині буде live-вивід coordinator-а, dispatch трьох ролей та їхні handoff-и. Він може
відрізнятися між запусками. Шукай порядок `test-author → implementer → reviewer`; текст усе одно
є коментарем до роботи, а не доказом результату.

Після кожного завершеного ходу Ralph сам повторно запускає:

```text
npm run lint && npm run test:fast && npm run links:check
```

Стабільний підсумок виглядає так:

```text
input-validation: 1 done, 4 todo · гейт зелений
```

Якщо `done` не зросло, лічильник плато рухається до `NO_IMPROVEMENT=2`. Якщо гейт червоний двічі
поспіль, спрацьовує `K_FAILURES=2`. Після двох дозволених ходів незавершена фіча зупиниться так:

```text
=== СТОП: ліміт ітерацій (MAX_ITER=2) вичерпано без DONE (ітерацій: 2) ===
```

Це штатна зупинка, а не обов'язково помилка.

`MAX_ITER=2` означає два завершені або перервані task-checkpoints, а не два звернення до моделі.
Звичайний хід використовує чотири контексти: coordinator і три субагенти. Findings reviewer-а
можуть додати повторні виклики implementer-а та reviewer-а.

## Крок 5 — запусти Ralph

Покажи ведучому сухий прогін. Після дозволу запусти ту саму команду без `--dry-run`:

```bash
MAX_ITER=2 K_FAILURES=2 NO_IMPROVEMENT=2 npm run ralph -- --feature input-validation
```

Поки вона працює, нічого не вводь у цей термінал. Для аварійної зупинки натисни `Ctrl+C` один
раз і дочекайся повідомлення `Перервано на ітерації ...`. Після ручної зупинки не запускай цикл
повторно без ведучого: агент міг залишити незакомічений тест або код.

## Крок 6 — перевір факти після запуску

Спочатку подивись реальний стан задач:

```bash
cat docs/features/input-validation/tasks/tracker.md
```

Ralph рахує метрику безпосередньо з колонки `Status`. Текст агента «готово» не змінює `todo` на
`done`.

Тепер подивись локальні коміти:

```bash
git log --oneline --decorate -6
```

Одна завершена задача має дати один новий коміт. `git push` Ralph не виконує.

Подивись самозвіт агента:

```bash
tail -n 40 loop/JOURNAL.md
```

І нарешті виведи саме той блок пам'яті, який хук передасть наступній Claude-сесії:

```bash
printf '{}' | RALPH_FEATURE=input-validation node .claude/hooks/loop-memory.mjs
```

У блоці `Факти` будуть поточна гілка, чистота дерева, трекер і останні коміти. Нижче буде
`Журнал попередніх ходів`. Якщо журнал каже «T2 зроблено», а трекер каже `T2 todo`, вір трекеру:
факти обчислені з диска, журнал написала модель.

## Крок 7 — фінальні ворота

Прибери службовий сигнал `DONE`, якщо агент устиг його створити. Код і коміти ця команда не
чіпає:

```bash
node -e "require('fs').rmSync('DONE',{force:true})"
```

Запусти повну перевірку:

```bash
npm run verify
```

Очікуваний фінал:

```text
verify OK — 7 ✓
```

На цьому вправа закінчена. Не запускай наступні ітерації без ведучого.

## Альтернатива — Claude Code без Ralph

Одна Claude Code сесія доречна для довгої роботи з чіткою умовою завершення та циклом перевірки.
Тут такою умовою є: усі T1–T5 мають `done`, `npm run verify` зелений, робоче дерево чисте.

Після кроку 1 замість dry-run і запуску Ralph:

1. відкрий [GOAL_PROMPT.md](./GOAL_PROMPT.md);
2. скопіюй звідти весь блок, що починається з `/implement input-validation`;
3. встав його в Claude Code і запусти;
4. переконайся, що Claude назвав skill `implement`, а перший checkpoint почав із dispatch
   `test-author`, не з прямого редагування продукт-коду;
5. дивись статус у поточній Claude-сесії, а факти — через tracker, git і `npm run verify` з
   кроків 6–7.

Очікуваний порядок для кожної задачі:

```text
Claude coordinator
  → /implement input-validation
  → test-author: RED і точний failing assertion
  → implementer: GREEN → REFACTOR → GATE
  → reviewer: незалежний REVIEW_CLEAN
  → coordinator: tracker → done, COMMIT, checkpoint
```

Агенти працюють послідовно в одній гілці: наступний стартує лише після handoff попереднього.
Якщо coordinator сам пише тест або продукт-код до dispatch `test-author`, delegated-режим не
активувався — зупини сесію, а не продовжуй інший за задумом експеримент.

Claude Code замінює Ralph **за результатом**, але не копіює його внутрішню будову:

| | Ralph | Claude Code |
|---|---|---|
| Оркестрація | `ralph.mjs` запускає нового coordinator-а на хід; він викликає три project agents | `/implement` веде DAG і послідовно викликає ті самі три project agents |
| Прогрес | `done` у tracker після кожної ітерації | `done` у тому самому tracker після checkpoint |
| Контекст | нова Claude-сесія щоходу | довгий coordinator-контекст + чистий контекст кожного субагента |
| Пам'ять | git + tracker + `JOURNAL.md` через hook | поточна сесія + ті самі факти, handoff-и й checkpoint-журнал |
| Завершення | файл `DONE` або жорстка зупинка | перевірна умова з implementation-промпту |
| Результат | TDD, локальні коміти, зелені ворота | ті самі TDD, коміти й ворота |

Ralph корисний, коли треба продемонструвати холодний старт і зовнішню оркестрацію. Claude Code
коротший операційно: prompt задає умову завершення, `/implement` володіє протоколом, а іменовані
субагенти ізольовано виконують RED, GREEN і незалежне review.

## Що змінилося порівняно із задачею 1

| | Housekeeping | Ralph |
|---|---|---|
| Джерело роботи | `TODO/FIXME`, issues, червоні ворота, доказові спрощення | перша доступна `todo` у SDD-трекері |
| Промпт | `practice/task-1-housekeeping/PROMPT.md` | `loop/PROMPT.md` + динамічна ціль |
| Хто комітить | агент, один локальний коміт | агент за TDD-протоколом |
| Пам'ять | Git-історія між незалежними проходами | `JOURNAL.md` + свіжі факти від хука |
| Червоний гейт | скрипт робить відкат | наступний хід може виправити; після K провалів — стоп |

Ralph навмисно запускає новий процес агента для кожної ітерації. Довгий чат може накопичувати
застарілі припущення; холодний старт змушує щоразу перечитувати актуальний git, трекер і
специфікацію. Повний технічний опис — у [loop/README.md](../../loop/README.md).
