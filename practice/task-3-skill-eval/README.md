# Практика 3 — eval-и для скілів

У перших двох практиках результат роботи агента перевіряли готові ворота репозиторію. Тепер
подивимося на два рівні перевірки самого скіла:

- **3A — contract eval** (цей файл): важлива обіцянка з `SKILL.md` стає перевіркою з кодом
  виходу `0` або `1`, без виклику моделі;
- **3B — behavioral eval** (`behavioral/README.md`): справжній `review-only` запускається на
  golden task, а grader перевіряє відповідь і чистоту git.

Уся практика займає приблизно **45–60 хвилин**. Почніть із 3A: вона пояснює складові eval-а без
недетермінованості, а 3B переносить ту саму конструкцію на живий coding agent.

| Термін | Що означає тут |
|---|---|
| Target | скіл або поведінка, яку перевіряємо |
| Case | конкретна вхідна задача й очікуваний результат |
| Grader | код, який вирішує PASS або FAIL |
| Fixture | маленький стабільний репозиторій для запуску case |
| Golden task | один зафіксований prompt із відомим правильним результатом |
| Negative control | навмисна поломка, на якій grader мусить почервоніти |

## Перед стартом

Практика розрахована на вже підготовлений клон. Якщо ви щойно клонували репозиторій, спочатку
виконайте розділи [«Встановлення»](../../README.md#встановлення) і
[«Перевірка середовища»](../../README.md#перевірка-середовища): `npm ci`, встановлення Chromium
командою для своєї ОС, а потім `npm run doctor` і `npm run verify`. Повторно завантажувати
залежності перед кожною підзадачею не потрібно.

## Підзадача 3A — перший contract eval

Це code-along на **20 хвилин**. Модель не викликається, токени не витрачаються, нові залежності
не потрібні. Наприкінці у вас буде `eval/lint.mjs`, який:

- зеленіє на чинному скілі;
- червоніє на навмисно зламаній копії тексту;
- містить один eval-кейс, який ви обрали й окремо змусили почервоніти самі.

> Це детермінований contract eval: він перевіряє, що важлива інструкція досі записана у скілі.
> Він ще не доводить, що модель завжди виконає цю інструкцію. Поведінковий eval потребував би
> запусків агента на однаковому завданні та окремого grader-а — для першої вправи це зайве.

## Що саме перевіряємо

Візьмемо вендорену копію скіла у відкритому форматі:
[`sdd-implement/SKILL.md`](../../.agents/skills/sdd-implement/SKILL.md). У ній є правило:

```text
RED first, always
```

Це не декоративна фраза: вона стисло закріплює заборону писати production-код до першого
червоного тесту. Якщо фраза зникне, eval вимагатиме review — правило могли видалити або лише
переформулювати. Точний пошук навмисно не вміє відрізнити ці два випадки.

Наш мінімальний eval має чотири частини:

| Частина | У цій вправі |
|---|---|
| Мішень | текст `SKILL.md` |
| Кейс | правило `RED first, always` мусить існувати |
| Grader | точний пошук фрази |
| Контроль | `--break-case=<index>` видаляє обрану фразу лише в пам'яті |

## Крок 1 — почніть з окремої гілки

Ця практика не залежить від результатів задач 1 і 2. Почніть із чистої `main`:

```bash
git status --short
git switch -c workshop/skill-eval main
```

Якщо перша команда щось надрукувала або гілка вже існує, зупиніться й покличте ведучого.

Переконайтеся, що правило справді є у скілі:

```bash
git grep -nF "RED first, always" -- .agents/skills/sdd-implement
```

Має знайтися рядок у `.agents/skills/sdd-implement/SKILL.md`.

## Крок 2 — створіть один eval-кейс

У редакторі створіть каталог `eval` і файл `eval/lint.mjs`. Набирайте код разом із ведучим:

```js
import { readFileSync } from 'node:fs';

const SKILL_FILE = new URL('../.agents/skills/sdd-implement/SKILL.md', import.meta.url);

const checks = [
  {
    name: 'TDD починається з RED',
    phrase: 'RED first, always',
  },
];

let skillText = readFileSync(SKILL_FILE, 'utf8');
let failures = 0;

for (const check of checks) {
  const passed = skillText.includes(check.phrase);
  console.log(`${passed ? '✓' : '✗'} ${check.name}`);
  if (!passed) failures += 1;
}

console.log(`${checks.length - failures}/${checks.length} checks passed`);
if (failures > 0) process.exitCode = 1;
```

Тут немає AI-магії: файл читається як звичайний текст, а grader — це `includes`. Важливо інше:
назва кейса пояснює поведінку, яку ми захищаємо, а не лише повторює шукану фразу.

## Крок 3 — отримайте зелений результат

Запустіть eval із кореня репозиторію:

```bash
node eval/lint.mjs
```

Очікуваний результат:

```text
✓ TDD починається з RED
1/1 checks passed
```

Зелений запуск доводить лише сумісність із поточним текстом. Ми ще не знаємо, чи eval узагалі
здатен помітити поломку.

## Крок 4 — доведіть, що eval може почервоніти

Одразу після читання `SKILL_FILE` додайте контрольовану мутацію. Її значення — індекс кейса,
який треба зламати:

```js
const breakArg = process.argv.find((arg) => arg.startsWith('--break-case='));
if (breakArg) {
  const index = Number(breakArg.split('=')[1]);
  const check = checks[index];
  if (!Number.isInteger(index) || !check) {
    console.error(`Невідомий --break-case: ${breakArg}`);
    process.exit(2);
  }
  skillText = skillText.replace(check.phrase, '');
}
```

Вона не редагує файл на диску — лише прибирає одне обране правило зі змінної поточного
процесу. Зараз у масиві один кейс, тому його індекс — `0`.

На macOS, Linux або Windows запустіть ту саму команду:

```bash
node eval/lint.mjs --break-case=0
```

Тепер очікуємо навмисний провал і код виходу `1`:

```text
✗ TDD починається з RED
0/1 checks passed
```

Після нього ще раз виконайте звичайний `node eval/lint.mjs`: результат знову має бути зеленим.
Якщо обидва запуски зелені, перевірка нічого не захищає. Якщо обидва червоні, вона не описує
чинний контракт скіла.

## Крок 5 — додайте власний кейс

Знайдіть у `SKILL.md` ще одну інструкцію, втрата якої справді змінить поведінку. Для першої
спроби можна взяти заборону послаблювати тест:

```bash
git grep -nF "never make a test less strict" -- .agents/skills/sdd-implement
```

Додайте до масиву `checks` другий об'єкт. Назву сформулюйте своїми словами, а в `phrase`
скопіюйте точну опорну фразу зі скіла:

```js
{
  name: 'червоний тест не послаблюють заради GREEN',
  phrase: 'never make a test less strict',
},
```

Тепер звичайний запуск має показати `2/2 checks passed`. Далі доведіть, що **кожен** кейс
справді здатен почервоніти окремо:

```bash
node eval/lint.mjs --break-case=0
node eval/lint.mjs --break-case=1
```

Обидва запуски мають показати `1/2 checks passed` і завершитися з кодом `1`: у першому падає
готовий кейс, у другому — саме ваш. Синтаксис однаковий у Bash, zsh і PowerShell.

## Крок 6 — фінальна перевірка

```bash
node eval/lint.mjs
node eval/lint.mjs --break-case=0  # очікуваний exit 1
node eval/lint.mjs --break-case=1  # очікуваний exit 1
npm run verify
git diff --check
```

Готово, якщо eval зелений на справжньому скілі, **кожен** кейс окремо червонів через
`--break-case`, а ворота репозиторію лишилися зеленими. Зафіксуйте вправу окремим локальним
комітом:

```bash
git add eval/lint.mjs
git commit -m "test(skill): add first deterministic eval"
```

Не додавайте цей eval до `npm run verify` під час воркшопу. У репозиторії вже є сильніший
production-аналог — `npm run tools:check`: він перевіряє три погоджені інваріанти у чотирьох
вендорених копіях скіла. Тут ми навмисно відтворили один вузький зріз із нуля, щоб побачити
будову eval-а й додати власний кейс, а не дублювати готові ворота ще одним скриптом.

## Підзадача 3B — behavioral eval

Після contract eval відкрийте [behavioral/README.md](./behavioral/README.md). Там та сама
будова `target → case → grader → negative control` застосована до реального запуску
`review-only`: оцінюється вже не текст інструкції, а поведінка агента.

Після обох підзадач [практика 4](../task-4-self-improvement/README.md) використає той самий
принцип RED → GREEN, щоб приймати або відхиляти зміни housekeeping prompt.
